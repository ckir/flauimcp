# FlaUI.Mcp v0.7.2 — Typed-text verification for `desktop_type` (design)

**Date:** 2026-07-01
**Status:** design — approved shape, pending spec review
**Predecessor:** v0.7.1 (inter-key pacing). Supersedes the "reactive-editor non-keystroke path"
framing (Phase 4b.2) with an empirically-grounded, much smaller scope.

## 1. Problem

`desktop_type` drives text via `SendInput` (`KEYEVENTF_UNICODE`, paced 15 ms by default). Into the
**new Win11 Notepad** (a WinUI 3 `RichEditBox` with async writing-tools autocomplete) it garbles
multi-word input **non-deterministically and silently** — the caller gets `ok:true` with corrupted
text in the field. The garble reproduces at both 0 ms and 15 ms pacing, so keystroke pacing cannot
fix it.

## 2. Key finding that scopes this release (probed live 2026-07-01)

The new Notepad's Document element **exposes UIA `ValuePattern`** (`[Value,Scroll,Text]`).
`desktop_set_value` (UIA `ValuePattern.SetValue`, **lease-exempt**, already shipped) wrote the exact
multi-word spaced string that `SendInput` garbled — **byte-for-byte, verified by TextPattern
read-back**. So the reliable non-keystroke path **already exists and ships**. v0.7.2 does **not**
add a clipboard-paste tool or a LegacyIAccessible fallback.

The remaining gap is **silent failure + discoverability**: nothing tells a caller that a
`desktop_type` was corrupted, nor that `desktop_set_value` is the fix. v0.7.2 closes that gap.

## 3. Goal

Make `desktop_type` **optionally read back what it typed and report a mismatch as a soft,
actionable signal** that names `desktop_set_value` as the remedy — without ever crashing a step
that legitimately succeeded. Plus documentation steering reactive/RichEdit editors to
`desktop_set_value`.

## 4. Non-goals (explicit)

- **No hard error / throw** on mismatch (a legitimate app text-mutation must not crash the step).
- **No auto-retry** and **no auto-correction** of the field (already-typed text is left in place,
  matching the established partial-text-on-abort philosophy).
- **No append-case verification** (caret-at-end `before + typed`) — v1 verifies only the
  empty-field precondition; append is a documented future option.
- **No new tool** (`desktop_clipboard_paste`), **no LegacyIAccessible** path — moot per §2.
- **`settleMs` is not exposed** as a tool parameter in v1 (internal constant).

## 5. Design

### 5.1 Pure verifier core — `TypedTextVerifier` (headless-unit-tested)

The correctness logic is a **pure static helper** (no UIA, no `SendInput`, no sleep) so it runs on
the RDP-only test box where synthetic input cannot. Mirrors `UnicodeKeyTyper`.

```csharp
public enum VerifyStatus { Match, Mismatch, Skipped }

public readonly record struct VerifyOutcome(
    VerifyStatus Status,
    string? Reason,        // null on Match/Mismatch; a skip-reason token on Skipped
    string? Expected,      // populated on Mismatch only (the normalized typed text)
    string? Actual);       // populated on Mismatch only (the normalized read-back)

public static class TypedTextVerifier
{
    // before == null   -> couldn't read (TextPattern unsupported / read failed) -> Skipped("no-textpattern")
    // before != ""     -> field had content (RAW, pre-normalization)            -> Skipped("field-not-empty")
    // else compare normalize(after) vs normalize(typed) (Ordinal)              -> Match | Mismatch
    public static VerifyOutcome Check(string? before, string? after, string typed);

    // CRLF/lone-CR -> LF, then strip ALL trailing '\n'. Symmetric on both sides.
    internal static string Normalize(string? s);

    // Cap an echoed string at max chars; append '…' when it was longer. Pure/testable.
    internal static string Truncate(string s, int max);
}
```

**Normalization (both sides):** replace `\r\n` → `\n`, replace lone `\r` → `\n`, then strip **all**
trailing `\n` (recursive/symmetric — not just one). No other trimming (leading/interior whitespace
is significant). Stripping *all* trailing newlines prevents a false mismatch when the caller types a
trailing newline and the editor also appends its own structural newline (agy AGY-AFTER: double-NL
collapse).

**Precondition:** verification only *asserts* when the baseline field is the **empty string**
(`before == ""`, checked on the **RAW** value before normalization). This is deliberate: a truly
empty field has the caret unambiguously at offset 0 = end, so `after` must equal `typed`. A field
holding only a newline (`before == "\n"`) is **not** empty-precondition — its caret position is
ambiguous — so it is `Skipped("field-not-empty")`, avoiding the false mismatch agy flagged
(empty-before ambiguity). `before == null` is the "couldn't read" sentinel →
`Skipped("no-textpattern")`.

**Comparison:** `string.Equals(normalize(after), normalize(typed), StringComparison.Ordinal)` —
exact, no culture folding (text entry must be byte-exact).

**Reason tokens (open string, known values):** the pure verifier emits `"no-textpattern"`,
`"field-not-empty"`. The tool constructs the rest *as `VerifyOutcome(Skipped, …)` values that flow
through the same pure mapper* (so they stay headless-testable): `"read-failed"` (the after-read
threw — §5.3) and `"redacted"` (a before/after read came back `isPassword`/`[REDACTED]` — §5.4
security fold). `"disabled"` is a tool-layer `ran:false` case (no outcome; see §5.4). The set is
**open** — consumers must tolerate unknown reasons (§5.4 forward-compat).

### 5.2 Settlement delay

After the final keystroke, a reactive editor may still render uncommitted inline-prediction / IME
composition text; reading immediately can capture ghost text → false mismatch. The tool sleeps a
short **settlement delay** before the after-read. Internal constant `VerifySettleMs = 100` (ms).
Only incurred when verification actually runs.

### 5.3 Tool wiring — `InputTools.DesktopType`

New parameter, appended after `interKeyDelayMs` (keeps existing positional callers working):

```
verify (bool, default true) — read the element back after typing and report whether the
committed text matches what was typed. Soft signal only; never throws on mismatch.
```

**Default binding (agy round 2 — deser trap):** `verify` binds via the **same** MCP-SDK optional-
parameter mechanism as the existing `timeoutMs=4000` / `interKeyDelayMs=15` — an absent wire key
applies the C# default (`true`), NOT `System.Text.Json`'s primitive-`bool` zero-value. This is the
already-shipped behavior those params rely on; no custom converter is introduced. The contract test
(§6) pins the C# default; absent-key→true is the same guarantee the existing defaults already have.

**Redaction (security):** each TextPattern read (`before`, `after`) is checked for the
`isPassword`/`[REDACTED]` sentinel the read path already returns; if either is redacted, the tool
constructs `VerifyOutcome(Skipped, "redacted", …)` and echoes nothing (§5.4).

Flow:
1. In the existing focus lambda (`RunOnRefForInputAsync`), when `verify` is true, additionally read
   the element's **baseline** text via TextPattern. On success → `before = text`; on
   `PatternUnsupported`/read error → `before = null`. Return `(target, before)`.
2. `KeyType(...)` — unchanged (paced `SendInput`).
3. If `verify`: `Thread.Sleep(VerifySettleMs)`, then read the element **after** text via a second
   `RunOnRefForInputAsync` TextPattern read. If that read throws → construct
   `VerifyOutcome(Skipped, "read-failed", null, null)` (never fail the type).
4. `TypedTextVerifier.Check(before, after, typed)` (or the read-failed outcome from step 3) → map
   via `VerifyResult.From(outcome)` → fold into the result.

**Timeout accounting (agy AGY-AFTER — back-compat):** verification latency (`VerifySettleMs` + the
two TextPattern reads) is **NOT** charged against the caller's `timeoutMs`. `timeoutMs` continues to
bound only the ref-resolution/focus + type, exactly as today; the after-read uses its own independent
read timeout (`VerifyReadTimeoutMs = timeoutMs`, applied to the *second* `RunOnRefForInputAsync`, not
subtracted from the first). Verification can never turn a successful type into a `Timeout`/failure.

**TOCTOU (agy AGY-AFTER — clarification, not a change):** the `before`-read in step 1 sits between
`el.Focus()` and the `SendInput` in step 2, so it lengthens the focus→send interval. This does **not**
weaken the anti-focus-steal guarantee: the atomic foreground/hit-test re-verify runs **inside the
`SendInput` leaf, on the send thread, immediately before the keystrokes** (after the before-read), so
a focus-steal during the before-read still aborts the send (`ElementDisappearedDuringAction`). The
before-read cannot cause `SendInput` to blast the wrong window.

### 5.4 Result contract (the DTO the tool returns)

The pre-existing keys are unchanged; a `verify` object is **always** present:

**Invariant (agy AGY-AFTER round 2 — wire-contract):** the three booleans `ran`, `verified`,
`mismatch` are **ALWAYS present** in every variant (never omitted), so a statically-typed client
never hits a missing-key deserialization error. Only the *string* keys are conditional.

```jsonc
// verify disabled by caller:
{ "ok": true, "pathUsed": "synthetic",
  "verify": { "ran": false, "verified": false, "mismatch": false, "reason": "disabled" } }

// ran, asserted, matched:
{ "ok": true, "pathUsed": "synthetic",
  "verify": { "ran": true, "verified": true, "mismatch": false } }

// ran, could not assert — reason ∈ {"field-not-empty","no-textpattern","read-failed","redacted"}
// (field had content / no readable TextPattern / after-read threw / sensitive-field redaction):
{ "ok": true, "pathUsed": "synthetic",
  "verify": { "ran": true, "verified": false, "mismatch": false, "reason": "field-not-empty" } }

// ran, MISMATCH (the discoverability payload):
{ "ok": true, "pathUsed": "synthetic",
  "verify": { "ran": true, "verified": false, "mismatch": true,
    "expected": "smoke test v0.7.2 …",     // normalized typed, truncated (see below)
    "actual":   "smoke 000000.cccccced …", // normalized read-back, truncated
    "recommendedFallbackTool": "desktop_set_value",  // STABLE machine key
    "remedy":   "Text was not entered correctly — the target may be a reactive/RichEdit editor that races synthetic keystrokes. Use desktop_set_value (UIA ValuePattern) for reliable text entry." } }
```

- **`ok` stays `true` on mismatch** (the keystrokes were sent; this is advisory).
- **Truncation ceiling (agy round 2 — off-by-one):** `expected`/`actual` are truncated so the final
  emitted string length is **≤ `VerifyEchoMax = 256`** INCLUDING the ellipsis — i.e. when the source
  exceeds the cap, emit the first `VerifyEchoMax − 1` chars + `…` (never 257). Bounds context-window
  blowout (safety, not cosmetics).
- **`recommendedFallbackTool` (STABLE machine key) vs `remedy` (opaque):** machine consumers branch
  on `recommendedFallbackTool` (a stable tool-name token). `remedy` is opaque human/LLM-readable prose
  whose exact wording is **not** guaranteed stable across minor versions and **must not** be parsed by
  machines.
- **`reason` is an OPEN string** (known values today: `disabled`, `field-not-empty`,
  `no-textpattern`, `read-failed`, `redacted`) — published in the MCP schema as `string`, **not** a
  strict `enum`. Consumers MUST treat unknown reasons as forward-compatible (a future minor release
  may add tokens without a breaking bump).
- **Sensitive-field redaction (security fold):** the TextPattern read path returns `[REDACTED]` /
  `isPassword:true` for password fields (same as `desktop_get_text`). When either the `before` or
  `after` read is flagged redacted/password, verify does **NOT** assert or echo — it returns
  `verified:false, mismatch:false, reason:"redacted"` with **no** `expected`/`actual`. Secrets never
  enter the result.
- **Key presence:** `ran`/`verified`/`mismatch` **always** present. `reason` is present **only** when
  `verified:false && mismatch:false` (disabled/skip/read-failed/redacted). `expected`/`actual`/
  `recommendedFallbackTool`/`remedy` are present **only** when `mismatch:true`. Consumers must treat
  the conditional string keys as optional.

## 6. Tests (all headless — RDP-only box cannot run `SendInput`)

**Pure `TypedTextVerifierTests`:**
- raw-empty `before==""` + `after == typed` → `Match`.
- raw-empty `before==""` + garbled `after` → `Mismatch`, `Expected`/`Actual` = normalized values.
- `before == null` → `Skipped("no-textpattern")`.
- non-empty `before` (e.g. `"x"`) → `Skipped("field-not-empty")`.
- **newline-only `before == "\n"` → `Skipped("field-not-empty")`** (RAW-empty precondition, not
  normalized-empty — the agy empty-before-ambiguity guard).
- CRLF: `after == "a\r\nb"`, `typed == "a\nb"`, `before==""` → `Match`.
- trailing newline: `after == typed + "\n"` → `Match`; **and `typed=="foo\n"`, `after=="foo\n\n"`,
  `before==""` → `Match`** (strip-ALL-trailing-`\n`, the double-NL-collapse guard).
- surrogate pair / emoji exact → `Match`; differing emoji → `Mismatch`.
- `Normalize` unit cases (lone `\r`, `\r\n`, single-vs-multiple trailing-`\n` all stripped).

**Pure `Truncate` tests:** `<=256` chars unchanged; `257` chars → emitted length **exactly 256**
(`255` chars + `…`, NOT 257 — the agy round-2 off-by-one); assert `result.Length <= VerifyEchoMax`
for an oversized input.

**Contract test (`InputToolsContractTests`):** `DesktopType` exposes `verify` with default `true`.

**Pure mapper `VerifyResultTests`:** feed each `VerifyOutcome` **and** the two tool-constructed
cases into `VerifyResult.From(...)` → assert the exact §5.4 JSON shape/keys for: `Match`
(`verified:true`), `Mismatch` (`expected`/`actual` truncated + `remedy`), `Skipped("field-not-empty")`,
`Skipped("no-textpattern")`, **`Skipped("read-failed")`** (the after-read-threw branch — routed
through the pure mapper so it is headless-tested, not left to live smoke), **`Skipped("redacted")`**
(password/sensitive read — assert NO `expected`/`actual` in the emitted JSON), and **`disabled`**
(`ran:false`). For every variant assert the three booleans `ran`/`verified`/`mismatch` are
**present**; for `Mismatch` assert `recommendedFallbackTool == "desktop_set_value"` and both echoes
`.Length <= VerifyEchoMax`. This is the seam that keeps the tool-layer folding logic headless-
testable; only the live UIA read itself is exercised in live smoke.

**Live smoke (manual, documented):** type into the new Notepad → expect `verify.mismatch:true` +
remedy; type into a classic Win32 Edit (Run dialog) → expect `verify.verified:true`.

## 7. Backward compatibility

- Additive param (`verify`, default true) — existing positional/named callers unaffected.
- No new throw path; `ok`/`pathUsed` unchanged. The only observable change for default callers is
  (a) an added `verify` object in the result, and (b) added latency when verifying:
  `VerifySettleMs` (100 ms) + two TextPattern reads. Callers needing the old fire-and-forget speed
  pass `verify=false`.
- A failed read-back never converts a successful type into a failure (§5.3 step 3).
- `verify` default binds like the existing optional params (absent wire key → `true`, §5.3), so no
  existing caller silently loses/gains verification by omission.
- Result-shape stability: `ran`/`verified`/`mismatch` always present; `reason` is an open string;
  `remedy` prose is explicitly non-stable while `recommendedFallbackTool` is the stable machine key
  (§5.4) — a future minor release can add reason tokens without breaking strict clients.
