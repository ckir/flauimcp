# FlaUI.Mcp v0.10.1 — "Trustworthy Hands / Act-Attribution" — Design Spec

**Status:** DRAFT (AGY-FIRST forks consulted + user-decided; awaiting AGY-AFTER panel + user approval gate)
**Base:** v0.10.0 (`db7d2cd`, unpushed bundle on `master`). Fast-follow patch.
**Author:** Opus main thread, 2026-07-04.

---

## 1. Goal

Make the server's *hands* **legible and attributable**. Two cohesive additions:

1. **T8 — element-identity audit trace.** Extend the existing synthetic-input audit line so each
   recorded action names the **resolved element's stable identity** (RuntimeId + AutomationId +
   ClassName + ControlType + bounds), not just its host window. A reviewer reading the audit log can
   tell *which control* was acted on, forensically, without the log ever carrying user content.
2. **Intent overlay (`--overlay`).** Opt-in: draw a **red rectangle on the target** for a
   configurable moment (~500 ms) *before* each mutative action, so a human watching the screen sees
   *what the agent is about to touch* before it happens. The overlay draws the **same bounds** T8
   records — the on-screen rect and (on the synthetic-input path) the audit record agree.

Together: you can *watch* an action live (overlay) and *audit* it after the fact (T8) — the "human
holding the keys" gets both a live and a forensic view of every act.

---

## 2. Scope (locked)

**IN (v0.10.1):**
- T8 element-identity fields on `ActionTarget` + `InputAudit.Record`, populated on the
  **synthetic-input path** (`InputGuard.Authorize` + `AuthorizeTextMutation`).
- `--overlay` / `--overlay-ms=N` intent overlay covering **ALL mutative element-targeted actions**:
  the UIA-pattern path (`InteractionTools`: invoke/toggle/set_value/expand/select/scroll,
  scroll_into_view, set_focus) **and** the synthetic-input path (`InputTools`:
  type/key/click/set_caret/select_text_range/paste_text), **and** coordinate actions
  (click_at/drag) via a crosshair at the point.

**OUT (deferred, decided):**
- **UIA-pattern audit trace → v0.10.2.** The UIA-pattern path (`InteractionTools` →
  `RunOnRefActionAsync`) writes **no** audit today. Adding a fresh audit sink there is its own bounded
  step. v0.10.1 extends only the *existing* synthetic-input audit. (User decision 2026-07-04: "phase
  it"; agy concurred.)
- **Phase 10 #1** (`list_windows` `wN` handle + HWND-recycle TOCTOU) → 0.11.0.

## 3. Design forks — resolved (AGY-FIRST consult `0cb93949`, user-decided)

| Fork | Decision | Rationale |
|------|----------|-----------|
| **E. Overlay coverage** | **All mutative element actions** (UIA-pattern + synthetic + coordinate) | User + agy: the human wants to *see every act*; overlaying only synthetic input would leave invoke/toggle (the bulk) invisible. |
| **E. Audit coverage** | **Phase** — T8 extends synthetic-input audit now; UIA-pattern audit → 0.10.2 | Keeps the patch tight; the UIA-pattern path has no audit sink today, so widening it is its own step. |
| **A. Render mechanism** | **A1** — layered, click-through, topmost, no-activate window; GDI red rect; dedicated STA thread + message pump; shown then destroyed | Artifact-free, multi-monitor-correct, steals no focus/input. (agy agreed; rejected A2 screen-DC XOR — flickery/racy.) |
| **B. Timing** | **B3** — `--overlay-ms=N`, default **500** blocking, `0` = off; whole feature off unless `--overlay` | Pre-act visibility is the point (blocking default), but CI/fast use can shorten or disable. |
| **C. Coordinate actions** | Crosshair box at the point | Coordinate acts lack structural verification; the overlay *is* the "is this the right spot?" check. |
| **D. Redaction** | **Allow-list** (agy counter, adopted) — audit logs ONLY `RuntimeId, AutomationId, ClassName, ControlType, BoundingRectangle`; **withhold all else** | Fail-closed: `Name`/`Value`/`HelpText`/`ItemStatus`/`LegacyIAccessible.Description` can hold user PII/secrets. An allow-list can't leak a property nobody vetted. |

**agy-flagged risks folded (verified true, baked into contracts below):**
- **UIA-tree pollution** — the overlay window is itself a top-level window → a UIA node in the *same
  process* that serves snapshots. Perception must filter it (§5.4).
- **STA / COM-pump safety** — the ~500 ms wait MUST be an async `await Task.Delay` executed **off the
  action STA**, never a thread-sleep on it (would freeze the COM pump). The action re-resolves
  atomically after the wait (§5.3).
- **Element volatility during the wait** — the target may move/vanish in 500 ms; the overlay is a
  best-effort *preview*, not a lock — the action's own fresh resolve is authoritative (§5.3).

## 4. Feature T8 — element-identity audit trace

### 4.1 Contract

`InputAudit.Record` gains the resolved element's identity, appended to the existing one-line format so
the log stays **back-compatible** (existing fields unchanged, in order; new fields appended).

**Wire shape (append-only):**
```
[flaui-mcp][input-audit] ts=<O> window=<hwnd> pid=<pid> process=<name> action=<verb> len=<n> \
    rid="<runtimeId>" aid="<automationId>" class="<className>" ctype="<controlType>" bounds=<L,T,W,H>
```
- New fields appear **only when an element identity was resolved** (the ref/selector paths). When
  `Element == null` (a window/coordinate action, or any action that never resolved a leaf) the trailing
  element fields are **STRICTLY OMITTED** — NOT emitted as `?` (SEAT-G fold). This guarantees the line
  is **byte-for-byte identical** to the pre-T8 output for the no-element case, satisfying INV-T8-1 and
  keeping the 384 existing headless tests' exact-match assertions green. The parser contract is
  "trailing element fields are present together or absent together."
- **`rid`** = the UIA `RuntimeId` rendered as a stable dotted/CSV string (e.g. `42.1376068.4.1`).
- **All numeric formatting uses `CultureInfo.InvariantCulture`** (SEAT-G fold): `bounds=L,T,W,H` (which
  can carry negative multi-monitor coords, e.g. `bounds=-1920,0,500,500`) and `rid` must not pick up a
  locale's alternate separators/sign glyphs. (`ts` already uses the invariant `:O` round-trip format.)
- **String fields MUST be quoted + escaped** (SEAT-4 wire-contract fold). `AutomationId`, `ClassName`,
  and `ControlType` can legitimately contain spaces (e.g. a WinForms `ClassName`, or a localized
  ctype), which would break a naïve space-delimited `key=value` parser. The appended string values
  (`rid/aid/class/ctype`) are wrapped in double-quotes with any internal `"`/backslash/newline escaped
  (a single shared `AuditQuote(string?)` helper; a null/empty value renders as `""`). `bounds` is
  numeric CSV (`L,T,W,H`), no quoting. The **existing** fields (`ts/window/pid/process/action/len`)
  are unchanged per INV-T8-1 — only the appended fields adopt quoting.
- **Allow-list ONLY** (§3 fork D): `RuntimeId, AutomationId, ClassName, ControlType, BoundingRectangle`.
  **NEVER** `Name`, `Value`, `HelpText`, `ItemStatus`, `LegacyIAccessible.Description`, or any other
  property. Missing/empty allow-listed values render as `""`, never throw.

### 4.2 Data model

`ActionTarget` extends from window-only to optionally carry element identity:
```
readonly record struct ActionTarget(
    nint Root, int Pid, string? ProcessName, string? WindowClass,
    ElementIdentity? Element = null);          // null on the window/coordinate paths

readonly record struct ElementIdentity(
    string RuntimeId, string? AutomationId, string? ClassName,
    string? ControlType, Bounds Bounds);        // Bounds = L,T,W,H ints
```
- Appending `Element` with a default keeps every existing `new ActionTarget(root,pid,proc,cls)` call
  site compiling unchanged (back-compat, mirrors the `FindQuery.IgnoreCase` append pattern from T1).
- `ElementIdentity` reads **only** allow-listed UIA properties; every read is fail-soft
  (`try/…ValueOrDefault`), consistent with `InputTargeting`'s existing defensive reads.

### 4.3 Capture point

`InputTargeting.ResolveElementTarget(win, el)` already holds the live `el` — it builds the
`ElementIdentity` from `el` there and attaches it. `ResolveRefTarget(window)` (window-level, no leaf)
attaches `Element = null`. The two `InputGuard` audit emitters
(`Authorize` line ~60/62, `AuthorizeTextMutation` line ~152) pass `target.Element` into `Record`.
No change to the decision pipeline (elevation/lease/deny-list/budget) — identity is **recorded**, never
**gated on**.

### 4.4 Invariants
- **INV-T8-1 (append-only):** existing audit fields unchanged in name/order; a pre-T8 log parser still
  reads `ts/window/pid/process/action/len`.
- **INV-T8-2 (allow-list):** the audit line contains no UIA property outside the §3-D allow-list, on
  any element including password/redacted controls. (Pinned by a `StringWriter`-sink test asserting the
  line for a `Secret` PasswordBox carries `aid/class/ctype/bounds` but **no** Name/Value substring.)
- **INV-T8-3 (no new failure mode):** an element whose allow-listed reads throw/return empty still
  audits (fields become `?`) — identity capture never turns a permitted action into a crash.

### 4.5 Tests

**Headless (`Category!=Desktop`) — the security-critical guarantees, testable without a live element.**
The allow-list is enforced **structurally**: `ElementIdentity` has fields for *only* the 5 allow-listed
props — it cannot physically hold `Name`/`Value`/`HelpText`. So the sink's redaction + shape are pinned
by constructing `ElementIdentity`/`ActionTarget` **values directly** (no live `AutomationElement`
needed — FlaUI's `AutomationElement` has no public ctor and isn't mockable, per the repo's real-WPF
Desktop-test convention):
- `InputAudit.Record` with/without `Element` → exact line shape (append-only, quoting/escaping,
  empty→`""`).
- **Redaction pin (INV-T8-2):** build an `ElementIdentity` whose values embed a would-be-secret string
  and an embedded quote/space → assert the emitted line contains only `rid/aid/class/ctype/bounds`,
  quoted+escaped, and carries no field named `name`/`value`/`help`.
- `AuditQuote` escaping unit tests (space, `"`, backslash, newline).

**Desktop (`Category=Desktop`, controller-run) — the live-element extraction, needs a real element:**
- `ElementIdentity` extraction reads only the allow-listed props off a real `TestAppFixture` element
  (e.g. the `Secret` PasswordBox) — asserts we populate from `RuntimeId/AutomationId/ClassName/
  ControlType/BoundingRectangle` and never touch `Name`/`Value`.

## 5. Feature — Intent overlay

### 5.1 Flag / config
- `ServerOptions` gains `bool Overlay` (`--overlay`) and `int OverlayMs` (`--overlay-ms=N`, default
  `500`; `0` disables even if `--overlay` present). Off by default → **zero** behavior/latency change
  for existing users.
- **`OverlayMs` is clamped to `>= 0` at parse time** (SEAT-C fold): a negative value (e.g.
  `--overlay-ms=-100`) is treated as `0` (disabled), never passed raw to `Task.Delay` (which throws
  `ArgumentOutOfRangeException` on a negative) — a malformed flag must not crash the action pipeline.
- README/SKILL document it as a **watch-the-robot** aid, not a security control (a determined agent can
  still act; the overlay makes acts *visible*, it does not gate them).

### 5.2 Mechanism (A1)
A dedicated **STA thread** owns a single reusable **layered, click-through, topmost, no-activate**
window:
`WS_EX_LAYERED | WS_EX_TRANSPARENT | WS_EX_TOPMOST | WS_EX_NOACTIVATE | WS_EX_TOOLWINDOW`. It draws a
red rectangle (a few-px border, hollow interior) at a given screen rect via GDI, runs a message pump,
and hides the window after each flash. The window is created **once** (lazy, on first overlay) and
reused — no per-action window churn.

**Threading mandates (SEAT-A fold):**
- The STA/message-pump thread MUST be started with **`IsBackground = true`** — a foreground pump thread
  would keep the process alive and hang MCP-server shutdown.
- Because the single window is shared, concurrent actions (two actions in their async delay at once —
  possible since the delay is off the STA) can race the show/hide. The overlay carries a **monotonic
  request token**: `Show(rect)` returns/records the newest token; a `Hide` applies **only if it still
  owns the current token**, so a stale hide from an earlier-finishing action never clears a later
  action's rect. (The rect simply moves to the newest requester — last-writer-wins is the intended,
  well-defined behavior for a single-window preview.)
- The real GDI overlay implements **`IDisposable`** (SEAT-F fold): `Dispose` marshals a shutdown
  (`PostQuitMessage`/`WM_QUIT`) to the STA pump thread and `Join`s it, then frees GDI resources. It is
  registered as a DI singleton so the host's container disposal tears it down — this prevents orphaned
  STA threads + leaked GDI handles when multiple `Server` instances are created and disposed **in one
  process** (e.g. the in-process xUnit test runner), which `IsBackground=true` alone does not solve.

### 5.3 Action integration (the seam)
An **`IActionOverlay` seam** is injected into the action wrappers. Default binding is a **no-op** unless
`--overlay` is on (DI picks the real GDI impl vs `NullActionOverlay` from `ServerOptions.Overlay`) —
so Core/tests never depend on GDI.

**Ordering (critical — per agy STA/volatility catch + SEAT-2 post-auth catch):**
1. Resolve the target's bounds cheaply (from the already-resolved element on the action wrapper's STA,
   or the coordinate point).
2. **Run every authorization gate FIRST** — the overlay is invoked **post-authorization, pre-effect**.
   A refused action (deny-list / no-lease / elevation / read-only-mode / budget) MUST **never** flash a
   rect (SEAT-2 security fold: a refused act must not preview an action that won't happen, nor let a
   forbidden agent spam intent-rects). On the **synthetic path** this reuses the existing
   `InputGuard.PreflightInput` (non-consuming gate) → overlay+delay → the real effecting call
   (`KeyType`/`KeyChord`/… which re-runs the idempotent `Authorize` + budget-consume + audit + send);
   `desktop_paste_text` already uses exactly this preflight-then-authorize split. On the
   **UIA-pattern path** the gate is `GuardWrite` (read-only-mode) + a successful resolve; the overlay
   fires only after both pass, before the pattern call. On the **coordinate path** the gate is
   `InputGuard`'s hit-test/deny-list on the resolved root, likewise before the crosshair.
3. **Exit the STA**, then `await Task.Delay(OverlayMs)` while the overlay thread shows the rect. The
   wait happens on the async continuation, **not** on the action STA (COM pump stays live).
4. Perform the action, which **re-resolves atomically** on the action STA (the ref path re-resolves via
   `RunOnRefActionAsync`; the selector path re-walks fresh). If the element moved/vanished during the
   wait, the action's own resolve is authoritative — the overlay showed a best-effort preview; a stale
   preview never corrupts the act, and a vanished target fails through the normal
   `SelectorNoMatch`/stale-ref path.

> The overlay draws the **pre-wait** bounds; the act uses **post-wait** resolution. They can differ by a
> frame — acceptable for a *preview*. The overlay is explicitly **not** a correctness gate.

**Coverage:** the seam is invoked from BOTH `InteractionTools` (UIA-pattern) and `InputTools`
(synthetic) mutative wrappers, and from the coordinate wrappers (click_at/drag) with a point-crosshair.
Read-only tools never invoke it.

**GDI correctness mandates (SEAT-J fold):**
- The GDI `WM_PAINT` handler MUST `DeleteObject` **every** `CreatePen`/`CreateBrush`/etc. it creates
  (in a `finally`/`using`-style scope), and restore the DC's original object after `SelectObject`.
  Windows caps a process at ~10 000 GDI handles; a per-flash leak would silently accumulate and
  eventually crash the process with an uncatchable native fault, killing the MCP session. (A cheaper
  alternative worth considering at plan time: cache **one** pen/brush for the overlay's lifetime and
  reuse it, disposing in the `IDisposable` teardown — no per-paint allocation at all.)
- The seam **no-ops immediately if the resolved bounds have `Width <= 0` or `Height <= 0`** (UIA
  returns collapsed/offscreen elements with zero/negative dimensions). Feeding those to `SetWindowPos`
  or GDI draws can cause a full-screen paint, artifacts, or a P/Invoke error — a degenerate rect must
  simply skip the flash (the action still proceeds; overlay is a preview, INV-OV-4).

### 5.4 UIA-tree filtering (agy catch — mandatory)
The overlay window carries a **sentinel window-class** (e.g. `FlaUiMcpIntentOverlay`). The perception
layer (`desktop_list_windows`, snapshot roots, `PopupFinder.SearchRoots`) **excludes** any window whose
class equals the sentinel, so the overlay can never appear in a snapshot, be found by `desktop_find`, or
be targeted. `WS_EX_TOOLWINDOW` also keeps it off the taskbar/alt-tab. (Belt-and-suspenders: it is
hidden between flashes, so it is usually not even a live top-level window when a snapshot runs.)

### 5.5 Invariants
- **INV-OV-1 (opt-in, zero-default-cost):** with neither `--overlay` nor a non-zero `--overlay-ms`, no
  overlay thread/window is created and no action incurs added latency. (Pinned: `ServerOptions.FromArgs`
  test + `NullActionOverlay` is the default binding.)
- **INV-OV-2 (no STA block):** the delay is `await`ed off the action STA. (Design-enforced; documented
  in the seam's contract + reviewed.)
- **INV-OV-3 (not perceivable):** the sentinel-class window is filtered from every perception surface.
  (Pinned by a filter unit test on the window-list/scope-roots predicate.)
- **INV-OV-4 (preview, not gate):** overlay failure (GDI error, thread down) is swallowed and the action
  proceeds — the overlay must never break an act. (Pinned: seam impl catches + no-ops on draw failure.)
- **INV-OV-5 (post-authorization):** the overlay is invoked only *after* the action's gate passes and
  *before* the effect; a refused action flashes no rect. (Pinned by a fake-`IActionOverlay` recording
  seam in a **Desktop-category** test — it needs a real resolved element: a denied/no-lease action
  records zero overlay calls; an authorized one records exactly one.)
- **INV-OV-6 (reentrancy + lifecycle):** the shared overlay window is race-safe under concurrent
  actions via a monotonic request token (a stale hide never clears a newer rect), and its STA thread is
  `IsBackground=true` (never blocks shutdown). (Pinned headless by a request-token unit test on the
  token/hide logic — extracted as a pure helper so it needs no real window.)

### 5.6 Tests
- **Headless:** `ServerOptions` parsing (`--overlay`, `--overlay-ms=N`, `0`-disable, default 500);
  the perception sentinel-class filter predicate; `NullActionOverlay` is default when off; the seam is
  invoked by mutative wrappers and not by read-only ones (via a fake `IActionOverlay` recording calls).
- **Desktop / RDP-limited:** the actual GDI red-rect **cannot be visually CI-verified** on the
  headless RDP box (same limitation as synthetic input). A `[Category=Desktop]` smoke can assert the
  overlay thread/window **creates and tears down** without throwing; true visual confirmation is a
  controller **live-smoke** on a real console (documented, like the §6 coord-landing live-smoke in
  Phase 9).

## 6. Sequencing

**T8 → overlay** (T8 first: it defines the element-identity capture the overlay's bounds reuse on the
synthetic path). Suggested task order for the plan:
1. `ElementIdentity` + `ActionTarget.Element` + `InputAudit.Record` append (pure/headless, TDD).
2. Populate in `InputTargeting.ResolveElementTarget`; thread through `InputGuard` audit emitters.
3. `ServerOptions.Overlay/OverlayMs` + parsing (headless).
4. `IActionOverlay` seam + `NullActionOverlay` + GDI impl (STA thread/layered window) — Core seam pure,
   GDI impl Windows-only; DI binding by flag.
5. Wire the seam into `InteractionTools` + `InputTools` + coordinate wrappers (async ordering per §5.3).
6. Perception sentinel-class filter (§5.4).
7. Docs + version `0.10.0 → 0.10.1` (csproj `<Version>`/`<AppVersion>` + `installer/flaui-mcp.iss`),
   CHANGELOG `[0.10.1]`, ROADMAP, README "watch + audit" section, SKILL guidance, Known-limitations
   (overlay is visibility not a gate; RDP-headless can't visual-CI it; UIA-pattern audit is 0.10.2).

## 7. Known limitations (document at ship)
- Overlay is a **visibility aid, not an authorization gate** — it does not slow or block a determined
  agent's act beyond the opt-in delay; safety still lives in lease/deny-list/read-only-mode.
- Overlay's real rendering is **not CI-verifiable** on the RDP-headless box → controller live-smoke.
- **UIA-pattern actions leave no audit line** until v0.10.2 — only synthetic-input acts are audited
  (now with element identity).
- The audit's element identity is a **snapshot at resolve time**; RuntimeId is not guaranteed stable
  across app restarts (fine for forensics — it pins *which element in this session*).
