## #23 — CI builds the installer with Inno Setup 7, version-resiliently

### Problem
`release.yml` installs Inno via `choco install innosetup` and invokes a **hard-coded, version-pinned**
compiler path. Inno Setup 7 installs to `C:\Program Files\Inno Setup 7\` (64-bit Program Files, **not**
`(x86)` as v6 did), so the hard-coded path is brittle: if Chocolatey's `innosetup` package has rolled
to v7, the *next* release build breaks silently on a missing path. Local dev already uses Inno 7.

### Current state (verified)
- `release.yml:33` — `choco install innosetup --no-progress -y`
- `release.yml:36` — `& "${env:ProgramFiles(x86)}\Inno Setup 6\ISCC.exe" installer\flaui-mcp.iss`

### Design
- **CI keeps `choco install innosetup` UNPINNED** (installs the current release; today that is v7).
  *Decision (user, overriding agy's "pin to 7.x.x"):* do not pin the Chocolatey version. Accepted
  tradeoff: CI rides the latest Inno release, so a future v8 could in principle change `.iss`
  compilation semantics under us. This is judged low-risk for our simple script and preferred for
  simplicity; the version-resilient path (below) absorbs the version/bitness change either way.
- **Stop hard-coding the compiler path.** Discover it — simply and robustly (panel-hardened):
  ```powershell
  $found = @(Get-ChildItem "C:\Program Files*\Inno Setup*\ISCC.exe" -ErrorAction SilentlyContinue)
  if (-not $found) { throw "ISCC.exe not found under any 'Inno Setup*' directory — is Inno Setup installed?" }
  $found | ForEach-Object { Write-Host "Found ISCC: $($_.FullName)" }   # log ALL matches: a stray/extra install is then visible, not silent
  if ($found.Count -gt 1) { Write-Warning "More than one Inno Setup install found; using the first logged. Verify the version above." }
  $iscc = $found[0].FullName
  Write-Host "Using $iscc"
  & $iscc installer\flaui-mcp.iss
  ```
  Rationale for each guard (panel findings): the `Program Files*` glob covers the bitness change and
  `Inno Setup*` covers the version bump; **fail loudly** if nothing matches (a glob miss yields an empty
  `$found` → `& $null` would be a cryptic failure); **log every match** so a stray/extra install is
  visible in the CI log rather than a false-GREEN (a v6 compile of our `.iss` also exits 0); on a clean
  `windows-latest` runner there is exactly **one** Inno (choco just installed it), so `$found[0]` is
  unambiguous; the multi-install case is a theoretical anomaly that the log + `Write-Warning` surface
  for a human rather than silently mis-ranking.
  **Deliberately NOT used (panel):** (1) probing each candidate with `& ISCC /?` + a `[version]` cast to
  sort — measured `[version]$null` does not throw, but a failed probe yields a null sort key and an
  unreliable pick; over-engineered for a single-install runner. (2) `Sort-Object FullName -Descending`
  as a "pick highest version" — it is a **string** sort, so `Inno Setup 7` would sort above a future
  `Inno Setup 10` (`'7' > '1'`), silently selecting the older compiler. Auto-ranking versions is a trap
  here; take-first-and-warn is honest. We also do **not** hard-assert a version. **Decision (user, R3-3):** keep
  log-and-warn; a mandatory `major >= 7` CI guard was considered (it would make CI enforce "not v6"
  rather than rely on a human reading the log) and **explicitly declined** in favor of simplicity — the
  "silently builds with v6" scenario requires v6 to be pre-installed on `windows-latest`, which choco
  (installing the latest v7) does not create, so the log-and-warn visibility is judged sufficient.
- **Local dev** already has `ISCC.exe` on PATH (measured: `C:\Program Files\Inno Setup 7\ISCC.exe`), so
  a local pre-merge compile is simply `ISCC.exe installer\flaui-mcp.iss` — no Chocolatey needed. This
  local compile is the primary pre-merge proof; CI validation (below) is the release-time proof.

### Measurement (STOP-and-report fork, resolved during implementation)
**M-23:** Confirm `choco install innosetup` on the GitHub Actions `windows-latest` runner installs an
ISCC.exe that the `Get-ChildItem "C:\Program Files*\Inno Setup*\ISCC.exe"` glob finds. If choco installs
to a path the glob misses (e.g. `ProgramData\chocolatey\lib\...`), widen the glob or resolve the shim on
PATH instead. Do not assume the install location — verify against a real runner (or the choco package
layout) before relying on it.

### Testing / validation
No unit tests exist for CI and none can (the runner is the only place this executes). Validation is:
1. **Local:** `ISCC.exe installer\flaui-mcp.iss` compiles clean (already confirmed under Inno 7 this
   session — exit 0).
2. **CI dry-run:** `release.yml` **already declares `workflow_dispatch: {}`** (`release.yml:5` — "manual
   run: builds/tests/publishes to verify CI; skips the Release step (no tag)"), so no trigger needs
   adding. A manual dispatch on the branch runs the Install-Inno and Build-installer steps — i.e. it
   exercises exactly the #23 change — without tagging a release. The `Write-Host "Using …"` line makes
   the resolved compiler/version visible in that run's log, so the dry-run actually proves *which* Inno
   built the installer (not just that some build exited 0). Because v0.15.0 is HELD, this dispatch is
   the pre-lift proof; the eventual tagged release is the final proof.

### Contract
`release.yml` build step no longer references a version-specific directory. No C# change. No change to
the `.iss` file itself.

## #24 — Scope the uninstall backup sweep to the `--agent` selector

### Problem
On **any** `uninstall` (including `--agent claude`), the router sweeps timestamped `.bak-*` backups
next to `{AgyServers, AgyPerms, GenericPath}` unconditionally. Since the agy config paths resolve to
the real `~/.gemini/*`, `uninstall --agent claude` deletes the user's real agy backups — a config the
run did not touch. Pre-existing since ≤ v0.14.0 (not a v0.15.0 regression); surfaced by the Task 12
smoke deleting real `~/.gemini/*.bak-*`.

### Current state (verified)
- `CliRouter.cs:112` — `foreach (var line in SweepBackups(new[] { paths.AgyServers, paths.AgyPerms, paths.GenericPath }))`
- `CliRouter.cs:387` — `private static IEnumerable<string> SweepBackups(IEnumerable<string> configPaths)` — already a **pure, agent-unaware** function; matches only `<config>.bak-*` (own pattern), dedups via a `HashSet`.
- `Paths` record (`CliRouter.cs:208`) carries `AgyServers, AgyPerms, GenericPath` (among others).

### Design
- **Leave `SweepBackups(IEnumerable<string>)` untouched** — it stays a pure file utility; routing logic
  must not leak into it.
- Add a small **private helper** on `CliRouter`:
  ```
  private static IEnumerable<string> PathsForAgent(string agent, Paths paths)
  ```
  returning the config-path subset the sweep should target, mirroring the `Apply()` agent dispatch:
  | agent (case-insensitive) | swept config paths |
  |---|---|
  | `agy` | `AgyServers`, `AgyPerms` |
  | `generic` | `GenericPath` |
  | `claude` | *(none)* — Claude has no local backing config file (opaque `claude mcp` CLI) |
  | `all` | union of the above (AgyServers, AgyPerms, GenericPath) |
  | anything else | *(none)* |
- The uninstall call site (`CliRouter.cs:112`) passes `PathsForAgent(agent, paths)` instead of the
  hard-coded triple. `claude` → empty → `SweepBackups` is a natural no-op. `all` → the current
  behavior, unchanged. `SweepBackups`'s existing dedup keeps a doubled path harmless.
- **No new class** (`AgentResourceRegistry`) — YAGNI until a third consumer beyond {sweep, Apply
  dispatch} exists (agy concurred).

### Contract / edge cases
- **The omitted/default case must NOT skip cleanup (panel-verified).** `CliRouter.cs:16` sets
  `agent = OptionValue(args, "--agent") ?? "all"`, and the installer's `[UninstallRun]` calls
  `uninstall --agent all` explicitly (`flaui-mcp.iss:44,46`). So a bare `uninstall` or a full product
  uninstall resolves to `agent = "all"` → `PathsForAgent` returns the agy+generic union → backups are
  swept exactly as today. There is **no `null`/empty agent path** that could fall through to the
  no-sweep bucket and orphan `~/.gemini` backups. The `"all"` arm of `PathsForAgent` is therefore
  load-bearing and must include AgyServers, AgyPerms, and GenericPath.
- `agent` is the same value `Apply(agent, …)` consumes. The helper uses the same
  `Equals(…, OrdinalIgnoreCase)` / `all` semantics as `Apply` so the two never disagree about what
  "claude" or "all" means.
- Unknown agent → empty set. Consistent: `Apply` also configures nothing for an unknown agent, so
  sweeping nothing matches "we touched nothing." The helper must not throw or over-sweep on a surprise
  value.

### Maintenance coupling (panel)
`PathsForAgent` is a hand-maintained parallel to `Apply()`'s agent dispatch: a new agent added to
`Apply` but not to `PathsForAgent` would silently never sweep its backups. Mitigate by deriving both
from a single source of truth if practical (e.g. one place that enumerates `{agent → config paths}`),
or at minimum add a code comment at each site pointing at the other so the coupling is visible. The
plan should pick one of these; do not leave the two lists silently independent.

### Testing
- **Pure unit test** on `PathsForAgent` (the testable unit): `claude` → empty; `agy` → the two agy
  paths; `generic` → the generic path; `all` → all three; unknown → empty. (Access: make the helper
  `internal` + `InternalsVisibleTo` the test project if that pattern already exists in the repo, else a
  router-level observable test — decided in the plan.)
- **Router-level observable test** using `--config <path>` (which redirects AgyServers/AgyPerms/
  GenericPath all to that path — verified in `ResolvePaths`): seed `<config>.bak-<ts>` files, then
  assert `uninstall --agent claude` sweeps **zero** of them while `uninstall --agent all` sweeps them.
  This proves the real-`~/.gemini` protection without touching the real profile.
- **State-verify the one existing sweep test (panel).** `CliRouterTests.cs:78`
  (`Uninstall_sweeps_our_backup_files_next_to_the_config`) uses `--agent generic`; under #24 `generic`
  still sweeps, so it **still passes** (measured) — but the plan must open that file and confirm it,
  not assume, since this session already hit "a behavior change silently breaks an existing test"
  twice (Tasks 7 and 10). It is also the proof-of-concept that `--config` redirection isolates the
  sweep for the new tests above.

## #25 — Bound the cumulative `claude`-CLI wall-clock per pass

### Problem
Each `claude` invocation in `ClaudeCollisionRemedy.Apply/Restore` gets its own 30s `ProcessRunner`
timeout, but nothing bounds the **sum**. Worst case `(1 + 2N) × 30s ≈ 3.5 min` for `N=3` colliding
entries if `claude` hangs, with Setup running hidden and no cancel window. Not a correctness/data-loss
bug (still bounded, completes), but a poor hidden-installer experience. Review-flagged Low.

### Current state (verified)
- `ProcessRunner.DefaultTimeout` = 30s; `ProcessRunner.Run(file, args, cwd, timeout)` **already takes a
  timeout argument**. Sentinels: `NotFound = -1`, `TimedOut = -2`.
- The runner passed to the writer and the remedy is built once in the claude branch:
  `CliRouter.cs:255` `var runner = ClaudeRunner();`, factory at `CliRouter.cs:303`, which bakes
  `DefaultTimeout` into each `ProcessRunner.Run` call.
- `ClaudeCollisionRemedy.Apply/Restore` call the injected runner (no timeout parameter of their own):
  enumerate (`plugin list --json`), then per entry a `disable`/`enable` and, only on an ambiguous exit,
  a re-read.

### Design — closure-deadline (agy conceded this over its `CancellationToken` proposal)
- **Do not change the runner `Func<string,string[],string?,RunResult>` signature** (frozen this
  session; a token would ripple through the writer, the factory, and ~5 fake runners in tests).
- At the **start of the claude-branch pass**, create one deadline covering the whole pass:
  a total budget `ClaudeBudget` (proposed **120s**, tunable named constant). **Sizing rationale
  (panel):** the per-call 30s `DefaultTimeout` already bounds a *single* hang; this cumulative cap
  exists only to stop the `(1 + 2N) × 30s` pile-up when *several* calls hang. It must sit comfortably
  **above** `N_expected × DefaultTimeout + margin` (realistic `N ≤ 3` colliding copies; 120s = 4×30s
  gives a full call of grace over three) so process-start/teardown overhead on hung predecessors can
  never squeeze a *legitimately slow* later call below its needed time — a 90s cap would false-kill a
  slow call-4 after three 30s hangs ate ~90.3s. The cap only bites a genuine multi-call pile-up.
  Use `System.Diagnostics.Stopwatch` (a monotonic elapsed clock).
- The `ClaudeRunner` factory takes the budget and returns a closure that, per call, computes
  `remaining = budget - stopwatch.Elapsed` and:
  - if `remaining <= TimeSpan.Zero` → **short-circuit**: return `new RunResult(ProcessRunner.TimedOut, "")`
    without invoking `ProcessRunner` (guards agy's negative-timeout case);
  - else → `ProcessRunner.Run(file, args, cwd, min(DefaultTimeout, remaining))`.
- Because each call's timeout is pre-shrunk to fit the remaining budget, the **sum of waits cannot
  exceed the budget + one process-start epsilon** — the same physical guarantee as a token, with no
  in-flight abort needed (the loops are strictly sequential). The first call always gets the full
  `min(30s, budget)`, so the common single-entry path is never falsely squeezed.
- A budget-exhausted call returns `TimedOut`, which `ClaudeCollisionRemedy` already degrades into a
  "claude timed out" warning — correct conservative behavior (no state change claimed).

### Scope of the budget
One budget covers the **entire claude-branch pass** (the `mcp remove`/`mcp add` writer calls plus the
remedy's `Apply`/`Restore` calls all share the one deadline via the shared runner). The `Stopwatch`
measures **total install/uninstall branch wall-clock**, which includes the filesystem
`Deploy()`/`Remove()` step running between the CLI calls — this is deliberate and is the correct
quantity to bound: what stalls a hidden installer is total elapsed time, not CLI time alone. (An
earlier draft called this "total *claude* wall-clock", which mismatched the implementation; corrected.)
Normal `mcp` and filesystem calls are near-instant, so they consume negligible budget. The writer's
own budget-exhaustion path is already defined and non-crashing: `ClaudeCodeConfigWriter.Install/Uninstall`
map `TimedOut` → `AgentChange.Failed` (`ClaudeCodeConfigWriter.cs:32,54`), not an exception. (And a hung
`mcp add` cannot even drain the whole budget — the first call is capped at `min(30s, budget)`, so it
returns at its own 30s long before 120s.)

### Interaction with Restore's marker consumption
`Restore` consumes the collision marker after its loop even when an individual `enable` **fails**
(the settled R7 design: a known-state failure still consumes; only an inventory-read failure keeps the
marker). A budget-exhausted `enable` returns `TimedOut` = a failed enable, so **budget exhaustion
mid-Restore consumes the marker and leaves the user's plugin disabled** with a manual-recourse warning.
This is a *new trigger* for that existing path. Two requirements fall out: (a) the 120s budget must be
generous enough that a normal `N ≤ 3` Restore always completes well inside it (it is — 3 fast enables
plus one enumerate is seconds); (b) confirm the `TimedOut`-enable branch prints the manual re-enable
command (`claude plugin enable <id> --scope <scope>`), so a rare budget cutoff is recoverable by hand.

### Contract / edge cases
- Fake test runners (`FLAUI_MCP_FAKE_CLAUDE_MISSING`, `FLAUI_MCP_FAKE_CLAUDE_COLLISION`) return instantly
  and never call `ProcessRunner`, so the budget never affects them — no existing test changes behavior.
- The deadline must start at pass entry, not at process start, so a slow `dotnet`/JIT warmup before the
  first call does not eat the budget unfairly. (Start the `Stopwatch` inside the claude-branch lambda.)

### Testing
**The budget lives in the `ClaudeRunner` closure**, which the existing `FLAUI_MCP_FAKE_CLAUDE_*` seams
**bypass** (they never reach `ProcessRunner`). So the budget test must exercise the budget-closure
itself, driven by an **injectable clock** — not one of the ProcessRunner-bypassing fakes, which would
test nothing about the budget. Concretely:
- Unit test: advance the injected clock past the budget across several calls and assert (a) once the
  clock exceeds the budget the closure short-circuits to `TimedOut` **without** invoking the underlying
  process, and (b) the per-call timeout handed to `ProcessRunner.Run` equals `min(DefaultTimeout, remaining)`.
- Unit test: with the clock inside the budget, a single call is unaffected — it gets the full per-call
  timeout, no premature `TimedOut`.
- Exact injection shape (an `Func<TimeSpan>`/clock parameter on the factory vs a wrapping decorator) is
  decided in the plan; the requirement is that the budget-closure is reachable by a test without real
  sleeping. Prefer an injectable clock.

## Cross-cutting decisions

- **Packaging:** one spec, three sections → one plan with three independent task-groups; each task-group
  commits separately (bisectable; agy's rollback concern met without three separate specs).
- **Release vehicle:** all three fold into the **held v0.15.0** branch (#23 is required for v0.15.0 to
  release at all; #24/#25 harden v0.15.0's own install path). They do **not** constitute a separate
  release. The v0.15.0 hold stays in force until the user lifts it; this spec's work happens on the same
  branch.
- **Ordering:** independent; any order. Suggested: #24 and #25 (C#, unit-tested, self-contained) before
  #23 (CI, validated only by a run).

## Out of scope
- Changing `--purge-data` to honor `--agent` (separate pre-existing router semantics; not this).
- Any change to the collision detection/disable/restore *logic* (verified correct by the review).
- Retiring the marketplace (still user-deferred, tracked separately).
- Pinning the Chocolatey Inno version (explicitly declined by the user for #23).

## Open measurements (resolved in the plan, each a STOP-and-report fork)
- **M-23:** the GitHub Actions runner's `choco install innosetup` puts an ISCC.exe where the
  `Get-ChildItem "C:\Program Files*\Inno Setup*\ISCC.exe"` glob finds it (else widen the glob / use the PATH shim).
- **#24 test access:** whether the repo already has `InternalsVisibleTo` for the test project (drives
  pure-helper vs observable-only testing).
- **#25 clock injection:** whether to inject a time source for deterministic timing tests vs a real
  `Stopwatch` (prefer injectable).
