# Review: FlaUI.Mcp Distribution & Installation Design

**Date:** 2026-06-25
**Reviewer:** AI Collaborator

This review evaluates the proposed distribution and installation design for the FlaUI.Mcp server, specifically focusing on the feasibility of the proposed Claude Code plugin and CLI-based installation strategy.

---

## Part 1: Assumption Verdicts

1.  **Claude Code Git LFS Fetching:** **REFUTED / HIGH RISK**. Standard package manager implementations and ZIP-based downloads typically do not resolve Git LFS pointers, resulting in the download of small text files rather than the actual binary. Additionally, hosting large binaries via LFS on GitHub risks hitting bandwidth quotas.
2.  **Antigravity (agy) Configuration:** **UNKNOWN**. While the general mechanism of editing JSON configuration is standard, the specific file precedence (e.g., `~/.gemini/settings.json` vs `~/.gemini/antigravity-cli/settings.json`) requires empirical verification against the active CLI implementation.
3.  **`${CLAUDE_PLUGIN_ROOT}` Expansion:** **LIKELY REFUTED**. MCP specifications do not natively support environment variable interpolation in the `command` field. Clients typically rely on path resolution relative to the `.mcp.json` file location.

---

## Part 2: Exhaustiveness & Risk Assessment

* **SmartScreen & AV (Severity: CRITICAL):** Shipping an unsigned, self-extracting .NET executable that synthesizes keyboard/mouse input is highly likely to trigger Windows Defender/SmartScreen.
* **UIPI Restrictions (Severity: HIGH):** Windows User Interface Privilege Isolation will block the server from interacting with elevated applications if the agent itself is not elevated. The design lacks a strategy for handling this failure mode.
* **JSON Corruption (Severity: HIGH):** Using standard JSON serializers to mutate configuration files risks destroying user comments or formatting unless `JsonCommentHandling.Skip` and `AllowTrailingCommas = true` are explicitly configured.
* **State Drift (Severity: MEDIUM):** Independent installation paths (PowerShell vs. Plugin) risk creating two separate, unlinked instances of `flaui-mcp.exe` on a single machine.

---

## Part 3: Creativity & Recommended Architecture

The current reliance on LFS bundling and CI-commit-back cycles is brittle. I recommend the **"Bootstrapper" Plugin Pattern**:

1.  **Remove the binary from the plugin:** Eliminate Git LFS usage entirely.
2.  **Use a Bootstrapper:** Ship the plugin with a small batch file (`.bat`) or script.
3.  **Unified Resolution:** When called, the bootstrapper checks for the existence of the binary in a standard location (e.g., `%LOCALAPPDATA%\Programs\FlaUI.Mcp`) and triggers the universal installer if it is missing or outdated.

**Benefits:**
* **Zero LFS Bloat:** The repository remains lightweight.
* **Eliminate CI Loops:** CI only needs to upload assets to GitHub Releases; no back-commits to `master` are required.
* **Consistency:** Both manual and plugin-based installations point to the same binary, preventing version drift.
